# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Sous Chef MCP is a Model Context Protocol server for Claude Desktop that extracts structured recipe data from websites using schema.org JSON-LD markup. It builds verified shopping lists from exact ingredient data rather than LLM interpretation of page text. Runs over stdio transport.

## Commands

```bash
# Setup
bash setup.sh                        # Creates venv, installs deps, creates default pantry, verifies compilation

# Run (used by Claude Desktop, not typically run manually)
./venv/bin/python server.py

# Verify server compiles
./venv/bin/python -m py_compile server.py

# Install deps manually
source venv/bin/activate && pip install -r requirements.txt
```

There are no tests, linter, or formatter configured.

## Architecture

**Single-file server** (`server.py`) built on `mcp.server.fastmcp.FastMCP`. Everything lives in one file: Pydantic input models, helper functions, tool implementations, and the server entry point.

### Key layers in server.py:
- **File I/O helpers** (`_load_json`, `_save_json`, `_load_yaml`, `_save_yaml`) ŌĆö all data persistence
- **Recipe extraction** (`_extract_jsonld_recipes`, `_collect_recipes`, `_extract_recipe_data`) ŌĆö parses HTML for `<script type="application/ld+json">` tags, recursively finds Recipe-typed objects including nested `@graph` structures
- **Ingredient categorization** (`CATEGORY_RULES` dict, `SPICE_KEYWORDS` list, `_categorize_ingredient`) ŌĆö keyword-matching to assign ingredients to store sections (Protein, Dairy, Produce, etc.)
- **Ingredient grouping** (`_base_ingredient`, `_group_ingredients`) ŌĆö groups like ingredients from multiple recipes using longest-keyword-first matching from `_ALL_KEYWORDS`
- **Pantry staple detection** (`_is_pantry_staple`) ŌĆö fuzzy substring match against `config/pantry_staples.yaml`
- **Pydantic input models** ŌĆö one per tool, with validation (e.g., `RecipeUrlInput`, `ShoppingListInput`, `MenuFormatInput`, `FeedbackInput`)
- **Tool implementations** ŌĆö 17 tools prefixed `recipe_*`, registered via `@mcp.tool()` decorator

### Important patterns:
- **Context injection**: All tools use `ctx: Context` (type annotation required for FastMCP to inject it). Access lifespan state via `ctx.request_context.lifespan_context["key"]`.
- **Lifespan**: `app_lifespan` async context manager loads config and creates a shared `httpx.AsyncClient` at startup.
- **HTML output**: `recipe_format_menu` outputs HTML (not plain text) so Apple Notes renders headings, lists, and checkboxes. Uses `-apple-note-checkbox` CSS styles for checklist items.
- **Apple Notes export**: `recipe_export_apple_note` writes HTML directly to Apple Notes via AppleScript, bypassing Claude's text reformatting.
- **Logging**: Errors log to both stderr and `data/error.log` (WARNING+ level for the file handler).

### Data flow:
1. `recipe_get` fetches a URL, extracts JSON-LD, normalizes to internal format, logs to `data/history.json`
2. `recipe_build_shopping_list` fetches multiple URLs, categorizes and groups ingredients, separates pantry staples
3. `recipe_format_menu` orchestrates the full workflow: fetches recipes, builds grouped shopping list as HTML with checkboxes
4. `recipe_export_apple_note` writes the HTML to Apple Notes via AppleScript

### Config files:
- `config/sites.yaml` ŌĆö recipe site discovery config (base URLs, search patterns, category paths). Not required to extract from a URL.
- `config/pantry_staples.yaml` ŌĆö ingredients to list separately as "verify stock". Gitignored; generated by setup.sh.

### Data files (all gitignored, auto-created):
- `data/history.json` ŌĆö auto-logged on every `recipe_get` call
- `data/favorites.json` ŌĆö user-saved recipes with tags
- `data/exclusions.json` ŌĆö blocked recipes/ingredients with reasons
- `data/feedback.json` ŌĆö locally saved feedback (fallback when gh CLI unavailable)
- `data/error.log` ŌĆö persistent error/warning log

## Dependencies

Python 3.10+, mcp[cli], httpx, beautifulsoup4, pydantic, pyyaml. All in `requirements.txt`. Optional: GitHub CLI (`gh`) for feedback issues and self-update.
